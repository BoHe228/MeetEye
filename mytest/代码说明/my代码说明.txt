多模态会议软硬件预研 - 面试说明稿

项目背景：思必驰实习项目，时间 2026.03.31-2026.06.30。
面试定位：这是一个面向会议终端的多模态感知预研项目，不是单纯的人脸识别或目标检测 demo。核心目标是把视觉方位、人脸/人体跟踪、定向拾音、语音增强和 FunASR 转写串成一条可验证的会议感知链路。
项目展示形态：项目整理为 GitHub 工程仓库，仓库地址为 github.com/BoHe228/MeetEye。本地仓库 remote 指向 git@github.com:BoHe228/MeetEye.git。仓库中包含中英文 README、Demo 视频链接、运行方式、参数说明、WebUI 说明和核心模块文档，面试时可以作为工程落地和代码组织能力的补充证明。


1. 一句话介绍

我在思必驰实习期间做的是一个多模态会议软硬件预研项目，目标是在会议室场景中同时利用视觉和音频信息，实现“谁在什么方向、是否在发言、语音内容是什么”的端侧感知原型。

视觉侧主要负责鱼眼全景展开、多人检测跟踪、角度/扇区估计、FaceID 动态建库和 WebUI 可视化；音频侧的设计目标是结合视觉方位信息做定向拾音、语音增强，再接 FunASR 完成会议语音识别与转写。


2. 30 秒版本

这个项目面向会议终端场景，核心是做一个视觉 + 音频的多模态会议感知原型。视觉侧我基于鱼眼摄像头做 GPU 全景展开，用 YOLO 做姿态/人脸检测，用 HybridSORT/BoT-SORT 做多人跟踪，并根据全景坐标估计每个人的方位角和所在扇区；在人脸识别上接入 AdaFace embedding，设计动态 FaceID 库和 TrackID-FaceID 绑定策略，解决小人脸、低像素情况下 FaceID 抖动和误合并问题。

后续音频侧的思路是使用麦克风阵列进行声源定位和波束形成，通过视觉得到的人物方位辅助定向拾音，再接语音增强和 FunASR，把“人在哪个方向”和“说了什么”关联起来，形成会议场景下的多模态软硬件预研闭环。


3. 1-2 分钟版本

我在思必驰实习期间参与了一个多模态会议软硬件预研项目，项目目标是验证会议终端里视觉感知和语音感知的协同能力。具体来说，系统希望在会议室里识别参会人位置、跟踪目标、估计说话方位，并最终和定向拾音、语音增强、ASR 转写结合起来。

我主要负责视觉模态原型。输入是鱼眼摄像头画面，首先在 GPU 上做鱼眼到全景图的展开，然后把全景图切片送入 YOLO 姿态/人脸模型做检测。为了适配多人会议场景，我接入了 HybridSORT/BoT-SORT 做连续跟踪，并把检测结果映射到全景坐标下计算水平角和扇区。这样系统不仅知道画面里有人，还能知道每个人大致处在哪个方向。

在人脸识别部分，我接入 AdaFace 提取 512 维人脸 embedding，并设计了动态人脸库。因为会议摄像头下人脸通常比较小，像素低，直接依赖人脸 embedding 会出现两个问题：阈值低时不同人容易合并，阈值高时同一个人又容易被拆成多个 FaceID。所以我做了几件事：一是增加 bbox 尺寸过滤，避免过小人脸污染人脸库；二是把 FaceID 更深度地绑定到 TrackID 上，减少同一条轨迹中 FaceID 来回切换；三是做了 debug dump，把人脸 crop、embedding 和最终 FaceID 保存下来，再离线聚类和可视化分析，定位误合并和误拆分的原因。

音频侧的整体设计是用麦克风阵列做声源定位和波束形成，通过视觉侧给出的参会人方位来辅助选择拾音方向，然后经过语音增强抑制噪声和混响，最后接 FunASR 做实时或准实时转写。视觉侧提供“人在哪个方向、哪个 TrackID/FaceID”，音频侧提供“哪个方向有声音、说了什么”，两者融合后就可以支撑会议发言人定位、说话人聚焦和转写结果关联。

鱼眼摄像头

  → 全景切片 + YOLO 姿态/人脸检测

  → 波束形成 Beamforming
  → 语音增强 / 降噪 / 去混响
  → VAD 切分
  → FunASR 语音识别
  → 发言人位置 + 语音内容关联

项目工程形态可以补充这样讲：

这个项目不是只在本地跑通一个脚本，而是整理成了一个可展示的 GitHub 工程仓库。仓库里有中英文 README、演示视频、系统流程图、快速启动命令、参数说明和核心模块文档。运行方式上既支持本地 OpenCV 模式，也支持 FastAPI + WebUI 模式。WebUI 里通过 WebSocket 接收摄像头帧，通过 WebRTC/H.264 推送推理结果，同时通过 JSON WebSocket 输出每帧目标的 TrackID、方位角、俯仰角、距离、ReID 特征和 FaceID 信息。

工程优化上，鱼眼展开、YOLO 推理和 OSNet ReID 尽量放在 GPU 上；YOLO 支持 TensorRT engine 推理；全景切片批量推理减少小目标漏检；跟踪和角度计算保持轻量 CPU 逻辑。README 中也整理了 Demo 视频、性能参考、TensorRT 导出方式和关键参数，方便别人复现和评估。


5. 视觉侧我做了什么

5.1 鱼眼全景展开

会议室希望一个摄像头覆盖尽可能大的视场角，因此使用鱼眼摄像头。鱼眼图像直接做检测会有明显畸变，所以我做了鱼眼到全景图的展开。

实现上预先计算映射矩阵，把每个全景像素反查到鱼眼原图坐标，运行时在 GPU 上完成采样，减少 CPU 端图像变换开销。展开后的人物在全景图中更适合检测、跟踪和方位角计算。

这里需要注意，鱼眼画面的“正畸”和“全景展开”不是一回事。如果只是把鱼眼图像正畸成一张普通透视相机画面，输出通常会更像一个中心视角相机。为了避免黑边，算法往往会裁掉边缘区域；如果强行保留全部视野，又会出现大量黑边，并且边缘区域被严重拉伸。

所以对于会议室这种需要覆盖多人、多方向的场景，我没有把鱼眼图像简单正畸成单张普通画面，而是更倾向于做全景展开或多视角切片。这样可以尽量保留鱼眼相机的广视角信息，让边缘方向的人也进入后续检测、跟踪和角度计算链路。

面试时可以这样解释：

普通正畸更适合把鱼眼中心区域变成一张看起来正常的透视图，但它容易牺牲边缘视野；本项目更关心会议室 360 度或大视场覆盖，所以采用全景展开和切片检测，而不是只保留中心区域。

5.2 全景切片检测

全景图宽度比较大，直接整图推理会导致小人脸、小人体目标分辨率不足。因此系统把全景图横向切成多片，分别送入 YOLO 批量推理，再把检测框映射回全景坐标做合并和去重。

这样可以在不显著增加延迟的情况下提升远距离小目标的检测效果。

此外，项目还支持 recall boost 补漏检测。主模型用姿态模型时，背身、遮挡或者边缘区域的人可能没有稳定关键点，容易漏检。补漏检测会额外跑一个轻量纯检测模型，把 pose 模型漏掉的人框捞回来，再和主检测结果做 IoU 去重融合。对于没有关键点的补漏框，系统用框顶附近的点合成头部参考点，用来继续估计方位角和扇区。

5.3 多目标跟踪

检测模型只能给出单帧里有哪些人，但会议系统需要知道“这一帧的人”和“上一帧的人”是不是同一个人。因此我在检测之后接入了多目标跟踪模块，默认使用 HybridSORT，也保留了 BoT-SORT 作为可选方案。

Track 模块的输入是每一帧 YOLO 得到的人体检测框、置信度、关键点以及 OSNet 提取的 ReID 外观特征；输出是带有稳定 track_id 的目标列表。后续的角度计算、FaceID 绑定、WebUI 显示和音频说话人关联，都是围绕 track_id 展开的。

跟踪模块主要解决四件事：

- 把单帧检测结果串成连续轨迹，让同一个参会人在多帧中保持同一个 TrackID；
- 在检测短暂漏检、遮挡或置信度下降时，通过 Kalman 预测维持轨迹连续；
- 在多人接近、交叉、并排移动时，结合运动信息和外观特征减少 ID switch；
- 给后续 FaceID、方位角、扇区输出和音频 DOA 融合提供稳定的人物索引。

HybridSORT 的核心可以按“三阶段关联”来讲。

第一阶段是高分检测和已有轨迹匹配。系统先用 Kalman 预测每条轨迹在当前帧的位置，然后把高置信度检测框和轨迹做关联。关联代价不只是 IoU，还会结合 VDC 和 TCM。IoU 反映检测框和预测框的重叠程度；VDC 是速度方向一致性，用来判断目标运动方向是否合理；TCM 是置信度调制，用来惩罚检测置信度突然异常变化的匹配。如果开启 ReID，还会把 OSNet 外观特征的余弦距离加入代价矩阵，帮助区分外观不同的人。

第二阶段是 BYTE 风格的低分检测匹配。会议场景里，人被遮挡、背身或者处在画面边缘时，检测置信度可能会变低。如果只保留高分检测，轨迹会很容易断。BYTE 的思路是：高分检测先完成可靠匹配，剩下没有匹配上的轨迹再去尝试匹配低分检测。这样可以把一些被遮挡但仍然存在的人接回来，提高轨迹连续性。

第三阶段是 OC-SORT 风格的再关联。它会对第一阶段没有匹配成功的轨迹和检测再做一次补充匹配，并且更重视上一帧真实观测位置，而不是完全依赖 Kalman 预测框。这样可以降低预测漂移带来的错误关联。

在本项目中，跟踪还做了几个工程适配：

- ReID 特征不是在 tracker 内部重新裁图提取，而是在检测阶段用 OSNet 提前提取，然后通过 det['feature'] 传给 tracker，减少重复计算；
- 全景图是首尾相接的，人在最左边消失后可能从最右边出现，所以加入了左右边界穿越匹配，避免全景边缘导致 TrackID 断裂；
- 支持 coast-frames 续命机制，短暂漏检时可以临时输出 Kalman 预测框，几帧内目标重新出现就接回原轨迹；
- 对空帧、无 ReID 特征、低分检测阈值等情况做了兼容，避免实时系统中因为一帧异常导致跟踪器崩掉；
- 跟踪输出保留 bbox、confidence、keypoints、feature 等元数据，方便后续角度计算、人脸识别和 WebUI 绘制。

面试时可以这样总结：

TrackID 是这个项目里的“短期稳定身份”。它不等于真正的人脸身份，但它比单帧检测稳定得多。FaceID 更适合做长期身份识别，但在会议小人脸场景下容易受像素质量影响，所以我把 TrackID 作为主线，把 FaceID 绑定在 TrackID 上。这样系统先保证“同一个运动目标不要乱跳”，再逐步用人脸和音频信息补充身份。

5.4 方位角、俯仰角、距离和扇区估计

在全景图中，水平坐标和方位角近似一一对应。因此我根据全景图宽度和标定参数，把每个目标的关键点或检测框中心映射成水平角。

除了水平角，系统也会输出俯仰角和距离估计。俯仰角用于描述目标相对摄像头水平面的上下位置；距离估计主要利用双眼关键点像素间距和标定关系，在有效范围内给出一个近似物理距离。虽然距离估计不是这个项目的核心，但它能让输出从“平面检测框”升级成更接近会议空间感知的 3D 结果。

系统还支持把 360 度划分成多个扇区，例如 8 个扇区，每个扇区选一个代表目标输出。这对后续麦克风阵列很重要，因为音频侧的定向拾音也天然是按方向或扇区组织的。

5.5 动态 FaceID

FaceID 的目标不是替代 TrackID，而是给 TrackID 提供更长期的人脸身份信息。

我接入 AdaFace 模型提取人脸 embedding，然后动态建库：

- 新人脸满足质量条件时新建 face1 / face2 / ...；
- 后续帧与已有库做余弦相似度匹配；
- 每个 FaceID 保存多个高质量样本；
- 调试时保存 crop、embedding 和 metadata，方便聚类分析。

项目里还预留了一个轻量的视觉说话检测方案。它基于 MediaPipe FaceLandmarker 提取嘴部关键点，计算嘴部开合比 MAR，用于判断画面中的人是否可能在说话。这个方法不能替代音频 VAD，因为张嘴不一定有声、有声也不一定能看到嘴，但它可以作为视觉侧的 active speaker 辅助信号，和音频 DOA/VAD 结果一起融合。


6. 我遇到的主要问题和解决方法

6.1 小人脸导致 embedding 不稳定

会议摄像头通常离人比较远，人脸 bbox 可能只有几十像素。即使 resize 到 112x112，模型看到的也只是放大后的低细节图像。

表现是：

- 阈值低：多个人容易被聚成一个 FaceID；
- 阈值高：同一个人容易被拆成多个 FaceID；
- 同一个 TrackID 内 FaceID 可能来回变化。

解决方法：

- 增加 bbox 尺寸过滤，小于一定尺寸的人脸不进入 FaceID 计算；
- 区分 TrackID 和 FaceID，TrackID 负责短期连续，FaceID 负责长期身份；
- 强化 TrackID-FaceID 绑定，避免同一条轨迹中 FaceID 频繁切换；
- 对人脸 crop 和 embedding 做离线聚类，分析误合并和误拆分。

6.2 FaceID 和 TrackID 的关系

一开始 FaceID 更依赖 embedding 相似度，后来发现小人脸质量不稳定时，embedding 会波动。因此我把逻辑改成：

TrackID 一旦绑定 FaceID，优先沿用该 FaceID；
只有 TrackID 消失并释放绑定后，FaceID 才能重新分配；
同一帧同一个 FaceID 不允许被多个 Track 抢占。

这样可以避免一个 FaceID 在两个目标框之间来回闪烁。

6.3 Debug 体系

为了判断问题来自检测、裁剪还是 embedding，我做了调试保存：

- 保存检测框 bbox crop；
- 保存送入 AdaFace 的 aligned face；
- 保存 512D embedding；
- 保存 frame_id、track_id、face_id、bbox、score、yaw 等 metadata；
- 离线根据 embedding 聚类，生成 HTML、散点图、热力图和每类图片文件夹。

这个调试链路让我能直观看到：一个 cluster 里是否混了多个 FaceID，或者一个 FaceID 是否分裂到多个 cluster。


7. 音频侧应该怎么讲

虽然我当前主要做的是视觉侧，但这个项目的完整目标是多模态会议系统，所以面试时需要把音频链路讲清楚。

7.1 麦克风阵列和声源定位

会议终端一般会使用多麦克风阵列。多个麦克风接收到同一个声源时会有时间差，这个时间差可以用来估计声音方向。

常见思路：

- TDOA：Time Difference of Arrival，到达时间差；
- GCC-PHAT：常见的时延估计方法，对混响和噪声相对鲁棒；
- DOA：Direction of Arrival，声源到达方向估计；
- SRP-PHAT / MUSIC / ESPRIT：更系统的阵列声源定位方法。

在会议系统中，音频侧可以估计“哪个方向有说话声”，视觉侧可以估计“哪个方向有人”。两者一结合，就能提升说话人定位的稳定性。

7.2 定向拾音和波束形成

定向拾音的核心是波束形成。麦克风阵列不是简单把所有通道相加，而是根据目标方向对不同麦克风信号做延时补偿和加权，使目标方向的声音相干增强，其他方向的噪声被抑制。

可以这样解释：

视觉侧给出目标方位角 θ；
麦克风阵列根据 θ 计算每个麦克风的延时；
对多通道语音做延时补偿和加权求和；
增强该方向发言人的语音，抑制其他方向噪声和串扰。

常见算法：

- Delay-and-Sum Beamforming：实现简单，适合原型；
- MVDR Beamforming：在约束目标方向不失真的情况下最小化噪声功率；
- GSC：广义旁瓣抵消结构，可做自适应干扰抑制。

7.3 语音增强

会议室里常见问题有：

- 空调噪声；
- 键盘声、桌面敲击；
- 混响；
- 多人同时说话；
- 远场语音信噪比低。

语音增强的目标是提升 ASR 前端输入质量。可以包括：

- 降噪；
- 去混响；
- 回声消除；
- 语音活动检测 VAD；
- 单通道或多通道神经网络增强。

在这个项目中，视觉方位可以帮助语音增强做方向选择。例如系统知道当前目标人在 60 度方向，就可以优先增强这个方向的语音。

7.4 FunASR

FunASR 是语音识别部分，可以接收增强后的语音流，输出文字转写。会议场景下还需要考虑：

- 流式识别；
- VAD 切分；
- 标点恢复；
- 说话人分离或说话人归属；
- 与视觉 TrackID / FaceID 的关联。

本项目可以把视觉侧的 TrackID/FaceID 和音频侧的 DOA/ASR 结果关联起来：

视觉：track3 / face2 / 方位角 75°
音频：声源方向约 72° / 当前有语音 / ASR 文本
融合：face2 或 track3 正在发言，内容为 ...

FunASR 本身不是一个单一模型，而是一套工业级语音识别工具链和模型生态。它早期论文来自阿里巴巴达摩院 Speech Lab，当前官方文档署名为通义实验室，阿里巴巴集团，并在 ModelScope / GitHub 体系下开源维护。它的定位是把语音识别从“实验室模型”推进到“可训练、可微调、可部署、可服务化”的工程系统。

FunASR 里比较核心的模型和模块可以这样理解：

- Paraformer / Paraformer-zh：中文或中英场景下的主力 ASR 模型，属于非自回归端到端语音识别模型，推理效率高，适合会议、通话、字幕等中文生产级识别任务；
- Paraformer-zh-streaming：流式识别版本，适合实时字幕、实时会议转写，能够按音频 chunk 增量输出文字；
- FSMN-VAD：语音活动检测模型，用来判断哪些时间段有人声，把长音频切成有效语音片段，避免无效静音进入 ASR；
- CT-Transformer / ct-punc：标点恢复模型，把 ASR 输出的无标点文本恢复成更接近自然阅读的句子；
- CAM++：说话人相关模型，可用于说话人分离或说话人标签输出，会议转写中常用于区分“谁说了什么”；
- SenseVoiceSmall：多功能语音理解模型，支持语音识别、情感识别和音频事件检测，适合需要同时判断语音内容和语音状态的场景；
- Fun-ASR-Nano：LLM-based ASR，使用 SenseVoice 编码器和 Qwen 系列解码器，适合多语言、方言、带时间戳的高质量识别；
- Qwen3-ASR / GLM-ASR-Nano：偏大模型路线的多语言 ASR，适合追求更强上下文理解、多语言和方言能力的场景；
- emotion2vec：情感识别模型，可用于识别语音中的情绪状态。

工业中选择 FunASR 的原因主要有四点。第一，它是开源并支持私有化部署的，会议语音往往涉及隐私，不能完全依赖云端 API。第二，它把 VAD、ASR、标点、说话人分离、流式识别、部署服务这些模块整合得比较完整，工程接入成本低。第三，它提供 CPU/GPU、ONNX、Libtorch、TensorRT、WebSocket/API 服务等部署路径，适合从算法验证过渡到真实产品原型。第四，它支持训练和微调，可以在企业内部会议、行业术语、产品名、人名等领域数据上做适配，而不是只能使用通用模型。

我对 FunASR 的掌握可以这样说：

我已经掌握了 FunASR 的基本训练、微调和应用流程。应用侧可以用 AutoModel 调用 Paraformer、SenseVoice、FSMN-VAD、ct-punc、CAM++ 等模型，完成离线识别、流式识别、VAD 切分、标点恢复、说话人分离和结果结构化输出；训练侧了解它的 JSONL 数据格式、wav.scp/text 转换、训练集和验证集配置、torchrun / funasr-train-ds 启动方式、单卡和多卡训练、checkpoint 保存、模型平均、断点续训和微调后模型加载。对于会议场景，我可以把增强后的音频送入 FunASR，再把输出的时间戳、说话人标签和文本结果与视觉侧 TrackID / FaceID / 方位角做关联。


8. 多模态融合思路

视觉和音频各有优缺点：

| 模态 | 优点 | 缺点 |
|------|------|------|
| 视觉 | 能定位人、能跟踪人、能做人脸身份 | 小人脸、遮挡、背身、光照影响 |
| 音频 | 能判断是否发声、能转写内容 | 混响、噪声、多人串扰、方向分辨率有限 |

融合的关键是互补：

- 视觉提供候选人和方位；
- 音频提供是否说话和语音内容；
- 方位角作为视觉和音频的对齐桥梁；
- TrackID/FaceID 作为会议发言人的身份线索；
- FunASR 输出文本作为会议内容线索。

可以用规则融合做第一版：

如果音频 DOA 与某个视觉目标方位差小于阈值，
且该方向 VAD 检测到语音，
则认为该 TrackID 对应的人正在说话。

后续可以升级为概率融合：

score = α * 视觉方位匹配分数
      + β * 音频 DOA 置信度
      + γ * 人脸/Track 稳定性
      + δ * VAD/ASR 置信度


9. 面试官可能追问的问题

Q1：你这个项目一句话到底解决什么问题？

这个项目解决的是会议终端里的多模态感知问题。不是单纯做人脸识别，也不是单纯做目标检测，而是希望系统能知道“会议室里有哪些人、他们在什么方向、谁可能在说话、说了什么”。视觉侧负责鱼眼全景展开、检测、跟踪、方位角和 FaceID；音频侧负责 DOA、定向拾音、语音增强和 FunASR 转写。最终目标是把视觉位置和语音内容关联起来。

Q2：你自己主要负责哪一部分？

我主要负责视觉模态原型和多模态接口设计。具体包括鱼眼图像到全景图的展开、全景切片检测、YOLO 姿态/人脸检测、HybridSORT/BoT-SORT 跟踪、方位角和扇区估计、AdaFace 动态 FaceID 建库、FaceID 和 TrackID 绑定、WebUI 可视化，以及为后续音频链路输出 track_id、face_id、角度和扇区信息。

Q3：为什么要用鱼眼相机？

会议室希望一个摄像头覆盖更大的视场角，普通摄像头容易只能看到局部区域。鱼眼相机可以覆盖更多方向，适合多人会议场景。但是鱼眼图像畸变大，边缘人物变形明显，所以不能直接按普通相机画面处理，需要做全景展开、切片检测和角度映射。

Q4：为什么不直接把鱼眼图像正畸成普通画面？

普通正畸通常会得到一个中心透视视角。为了避免黑边，边缘区域很容易被裁掉；如果强行保留边缘，又会出现明显黑边和拉伸。会议场景里边缘方向也可能有人，所以我更倾向于全景展开或多视角切片，把大视场信息保留下来，再做检测、跟踪和方位角计算。

Q5：为什么全景图还要切片检测，不能整图检测？

全景图横向很长，整图缩放送入模型后，人脸和人体会变得更小，尤其是远处人和边缘人，检测效果会下降。切片检测相当于提高局部目标在模型输入里的分辨率，再把每个切片的检测框映射回全景坐标并合并去重。这样可以改善小目标检测，同时保留全局角度关系。

Q6：Track 模块为什么重要？

检测只告诉我当前帧有哪些人，但不知道这一帧和上一帧是不是同一个人。会议系统需要连续身份，所以必须做 tracking。TrackID 是后续所有模块的稳定索引：角度估计按 TrackID 输出，FaceID 绑定到 TrackID 上，音频 DOA 也可以通过角度和 TrackID 关联到某个人。

Q7：HybridSORT 的核心原理是什么？

HybridSORT 可以理解为运动预测、检测匹配和外观特征融合。每一帧先用 Kalman 预测轨迹位置，再把检测框和已有轨迹做匹配。第一阶段用高分检测匹配已有轨迹，代价里包括 IoU、速度方向一致性 VDC、置信度调制 TCM，如果启用 ReID 还会加入 OSNet 外观特征距离。第二阶段用 BYTE 思路，把低分检测拿来补匹配未匹配轨迹，减少遮挡和低置信度导致的断轨。第三阶段用 OC-SORT 风格再关联，减少 Kalman 预测漂移带来的漏匹配。

Q8：Kalman 在 tracking 里起什么作用？

Kalman 用来根据历史位置预测当前帧目标可能出现的位置。检测正常时，它帮助计算检测框和轨迹之间的匹配关系；检测短暂漏掉时，它可以让轨迹继续存在一段时间，等目标重新被检测到后接回原来的 TrackID。它解决的是短期运动连续性问题，但不能单独解决多人交叉和外观相似问题。

Q9：ReID 特征在 tracking 里起什么作用？

ReID 特征提供外观信息，比如衣服颜色、纹理和人体外观。只用 IoU 和运动信息时，两个人靠得很近或者交叉时容易 ID 互换；加入 OSNet ReID 后，可以用外观相似度辅助判断当前检测更像哪条历史轨迹。但 ReID 也不是万能的，如果画面低清、遮挡严重、两个人穿得很像，外观特征也会不稳定。

Q10：你项目里对 tracking 做了哪些工程适配？

第一，ReID 特征在检测阶段提前用 OSNet 提取，传给 tracker 复用，减少重复裁图和推理。第二，全景图左右边界是首尾相接的，所以做了边界穿越匹配，避免人从左边出去、右边进来时 TrackID 断掉。第三，加入 coast-frames 续命机制，短暂漏检时可以临时输出 Kalman 预测框。第四，对空帧、无 ReID 特征、低分检测阈值等异常情况做了兼容，保证实时系统稳定。

Q11：TrackID 和 FaceID 有什么区别？

TrackID 是短期连续身份，解决的是同一个运动目标在视频帧之间不要乱跳。FaceID 是长期人脸身份，解决的是这个人是谁。会议场景下人脸小、像素低，FaceID 会受图像质量影响，所以我不把 FaceID 作为唯一主线，而是用 TrackID 保证连续性，再把 FaceID 绑定到 TrackID 上作为长期身份补充。

Q12：为什么不用单纯人脸识别做身份？

因为会议场景下人脸通常比较小，摄像头又是广角/鱼眼，小脸 resize 后细节不足，embedding 会不稳定。直接用人脸相似度会出现两个问题：阈值低容易把不同人合并，阈值高又容易把同一个人拆开。所以我把 TrackID 和 FaceID 分开：TrackID 负责短期稳定跟踪，FaceID 作为辅助身份信息，并且通过绑定策略减少抖动。

Q13：你怎么解决 FaceID 在两个目标之间来回闪的问题？

我做了更深度的 TrackID-FaceID 绑定。TrackID 一旦绑定了某个 FaceID，就优先沿用这个 FaceID，而不是每帧都重新按 embedding 最近邻切换。同时增加 FaceID owner 约束，同一个 FaceID 在同一时间只允许被一个活跃 TrackID 占有，只有对应 TrackID 消失并释放后，FaceID 才能重新分配。这样可以减少一个 FaceID 在两个目标框之间闪烁。

Q14：为什么人脸聚类会出现阈值高拆分、阈值低合并？

这是 embedding 聚类里常见的阈值矛盾。阈值高时，要求相似度非常高，同一个人在不同姿态、光照、模糊程度下的 embedding 可能达不到阈值，就会被拆成多个 FaceID；阈值低时，不同人的小脸 embedding 可能也比较接近，就会被合并。当前摄像头下人脸像素小、细节少，这个问题会更明显。

Q15：你怎么验证 FaceID 问题来自哪里？

我加了 debug dump 机制，把检测框 bbox crop、送入 AdaFace 的 aligned face、512 维 embedding、frame_id、track_id、face_id、bbox 尺寸、置信度等信息保存下来。跑完后离线聚类，查看每个 cluster 里混了哪些 FaceID、每个 FaceID 分布在哪些 cluster，并导出图片文件夹和可视化图。这样可以判断问题是检测框太小、裁剪质量差、embedding 不稳定，还是绑定逻辑有问题。

Q16：为什么要做 bbox 尺寸过滤？

因为小人脸即使 resize 到模型输入尺寸，也只是把低细节图像放大，不能恢复真实纹理。这样的 embedding 很容易污染动态人脸库。所以我把过滤逻辑改成直接看 bbox 宽高，小于阈值的人脸不参与 FaceID 计算和入库，宁可少识别，也不要用低质量样本污染库。

Q17：视觉方位怎么计算？

鱼眼图像展开成全景后，水平坐标和水平角度近似对应。可以根据全景图宽度、视场角和坐标位置，把人体中心点或关键点映射成方位角。然后再把 360 度划分成多个扇区，每个目标都有角度和扇区。这为后续音频阵列做 DOA 对齐提供了统一坐标。

Q18：视觉方位怎么帮助音频？

音频阵列可以估计声源方向，但在混响、多人说话或噪声情况下会不稳定。视觉可以提供“哪些方向有人”作为先验，减少音频搜索空间。例如只在有人所在的扇区上做波束形成或说话人匹配，可以提高定向拾音和说话人归属的稳定性。

Q19：定向拾音的基本原理是什么？

多麦克风接收到同一声源会有到达时间差。对于一个目标方向，可以计算每个麦克风需要补偿的延时，把目标方向的语音对齐后相加，这样目标方向声音会增强，其他方向声音因为相位不一致会被削弱。这就是 Delay-and-Sum 波束形成的基本思想。更高级的 MVDR 会在保证目标方向不失真的同时最小化噪声功率。

Q20：TDOA、DOA、GCC-PHAT 分别是什么？

TDOA 是到达时间差，表示同一个声源到不同麦克风的时间差。DOA 是声源到达方向，目标是估计声音来自哪个角度。GCC-PHAT 是一种常见的时延估计方法，通过广义互相关和相位变换增强时延峰值，在噪声和混响环境下比普通互相关更稳健。会议阵列里通常先估计 TDOA，再转换成 DOA。

Q21：语音增强在这个系统里做什么？

语音增强是 ASR 前端。会议室有远场、混响、空调声、键盘声、多人串扰等问题，直接送 ASR 会降低识别准确率。增强模块可以做降噪、去混响、回声消除、VAD 和多通道增强。结合视觉方位后，可以优先增强目标方向的语音，再送入 FunASR。

Q22：FunASR 放在系统哪一层？

FunASR 在音频后端。前面先做拾音、增强和 VAD，把高质量语音片段送入 FunASR 转写。识别结果再和视觉侧的 TrackID/FaceID、方位角进行关联，最终输出“谁在说话、说了什么”。

Q23：FunASR 是什么，来自哪个公司？

FunASR 是阿里语音团队开源的工业级语音识别工具链和模型生态，早期论文来自阿里巴巴达摩院 Speech Lab，当前官方文档署名为通义实验室，阿里巴巴集团，并在 ModelScope / GitHub 体系下维护。它不只是一个 ASR 模型，而是包含 VAD、ASR、标点、说话人、部署服务和训练微调流程的一整套工具。

Q24：FunASR 里面常用模型分别做什么？

Paraformer / Paraformer-zh 是中文或中英 ASR 主力模型，适合离线识别；Paraformer-zh-streaming 用于流式识别；FSMN-VAD 做语音活动检测；ct-punc / CT-Transformer 做标点恢复；CAM++ 可用于说话人相关任务；SenseVoiceSmall 支持语音识别、情感识别和音频事件检测；Fun-ASR-Nano、Qwen3-ASR、GLM-ASR-Nano 属于更偏大模型路线的多语言或高质量识别方案；emotion2vec 可用于情感识别。

Q25：工业中为什么会选 FunASR？

第一，它开源并支持私有化部署，适合会议隐私场景。第二，它覆盖 VAD、ASR、标点、说话人、流式识别和服务化部署，工程链路比较完整。第三，它支持 CPU/GPU、ONNX、Libtorch、TensorRT、WebSocket/API 等部署路径，便于从原型到产品。第四，它支持训练和微调，可以适配企业内部术语、人名、产品名和会议数据。

Q26：你对 FunASR 掌握到什么程度？

我已经掌握 FunASR 的训练、微调和应用流程。应用侧可以用 AutoModel 调用 Paraformer、SenseVoice、FSMN-VAD、ct-punc、CAM++ 等模型，完成离线识别、流式识别、VAD 切分、标点恢复、说话人分离和结构化输出。训练侧了解 JSONL 数据格式、wav.scp/text 转换、训练集和验证集配置、torchrun / funasr-train-ds 启动、单卡和多卡训练、checkpoint 保存、模型平均、断点续训和微调后模型加载。

Q27：这个系统怎么判断“谁在说话”？

第一步视觉侧给出每个 TrackID 的方位角和扇区；第二步音频侧用麦克风阵列估计当前声源 DOA，并通过 VAD 判断是否有人声；第三步把音频 DOA 和视觉目标角度做匹配。如果某个 TrackID 的方位角和声源方向接近，并且该方向有语音，就可以认为这个 TrackID 对应的人正在说话。后续可以再结合 FaceID 和 ASR 文本输出“谁说了什么”。

Q28：视觉和音频融合时会遇到什么问题？

主要问题是时间同步、坐标对齐和不确定性。视觉有帧率和推理延迟，音频有缓冲和 VAD/ASR 延迟，二者时间戳需要对齐。坐标上，视觉角度和麦克风阵列 DOA 要使用同一个方向定义。还有就是视觉可能看不到脸，音频可能有混响或多人同时说话，所以融合时不能只用硬规则，后续可以做置信度加权或概率融合。

Q29：这个项目最大的难点是什么？

最大的难点是会议场景的输入质量不稳定。视觉侧有人脸小、低像素、遮挡、背身、鱼眼畸变和边缘拉伸；音频侧有远场、混响、噪声和多人串扰。因此不能只靠单个模型，需要通过工程链路和多模态先验来提升稳定性。我的工作重点就是把检测、跟踪、FaceID、方位估计和调试分析链路打通，为后续音频融合提供可靠视觉输入。

Q30：如果让你继续优化，你会做什么？

视觉侧我会继续提升小目标质量，比如优化相机位置和分辨率、提高切片策略、增加人脸质量评估、把 raw match face_id 和 final face_id 都保存下来做更细的 debug。跟踪侧会进一步分析 ID switch 场景，调 ReID 权重、track buffer 和边界匹配策略。音频侧会接入真实麦克风阵列数据，完成 DOA、波束形成、语音增强和 FunASR 的闭环验证，再做视觉角度和音频方向的时间同步与置信度融合。

Q31：现在视觉模型和听觉模型是分开的，为什么不做成一体化模型？

当前项目采用的是分体式多模态架构，也就是视觉侧和音频侧分别建模，再在结果层做融合。视觉侧负责检测、跟踪、FaceID、方位角和扇区；音频侧负责 DOA、定向拾音、语音增强、VAD 和 FunASR 转写。两边输出结构化结果之后，再通过时间戳、方位角、TrackID/FaceID 和声源方向做关联。

分体式方案的优点是工程上更可控。第一，每个模块都可以用成熟模型和成熟算法，比如 YOLO、HybridSORT、AdaFace、麦克风阵列 DOA、波束形成、FunASR，不需要从零训练一个大模型。第二，调试更清晰，如果结果错了，可以判断是检测错、跟踪错、人脸错、DOA 错还是 ASR 错。第三，部署更灵活，视觉和音频可以分开优化延迟和算力，也可以单独替换某个模块。第四，数据要求更低，不需要大量严格同步、逐帧标注的音视频多模态训练数据。

一体化模型的思路是把视频帧、人物区域、音频特征、声源方向甚至文本一起输入一个统一模型，让模型端到端学习“谁在说话、说了什么、和画面中哪个人对应”。它的优势是理论上可以学到更深的跨模态关系，比如嘴部运动和语音的同步关系、人物位置和声源方向的一致性、语音内容和说话人状态的关联。在 active speaker detection、音视频说话人分离、端到端会议理解这类任务上，一体化模型有潜力取得更好的整体效果。

但一体化模型的问题也很明显。它需要大量高质量音视频同步数据，需要标注谁在说话、说话时间段、人物框、说话人身份和文本内容；训练成本和部署成本更高；对时间同步非常敏感；模型出错后可解释性差，很难快速定位问题；同时端侧会议终端算力有限，部署一个大的一体化模型未必现实。

所以如果面试官问，我会这样总结：

这个项目当前阶段更适合分体式架构，因为它是会议终端的软硬件预研，需要快速验证整条链路是否可行。分体式方案便于工程落地、调试、替换和部署；一体化模型是后续可以探索的方向，尤其适合在已经积累了足够音视频同步数据之后，用来提升 active speaker detection、说话人归属和跨模态鲁棒性。我的理解是：早期原型和工业落地先用分体式系统保证可控性，后续再用一体化模型提高上限。

Q33：开源仓库怎么体现你的工程能力？

它体现的是完整工程链路整理能力。仓库不是只放一个 notebook，而是包含本地模式、WebUI 模式、配置参数、模型路径、全景映射文件、核心模块拆分、调试文档和 Demo 视频。别人可以从 README 看到系统怎么跑、输出什么、参数怎么调、性能大概是多少。对实习项目来说，这能说明我不只是把模型跑通，也做了工程化封装、文档整理和可复现展示。

Q34：WebUI 在这个项目里起什么作用？

WebUI 是实时演示和调试入口。摄像头端通过 WebSocket 把图像帧推到 GPU 服务器，服务器完成鱼眼展开、检测、跟踪、角度、人脸等推理后，把标注视频通过 WebRTC/H.264 推给浏览器，同时用 JSON WebSocket 输出结构化结果。这样面试或演示时可以直接在浏览器看到实时画面、目标框、TrackID、FaceID、角度和性能状态。

Q35：为什么要用 WebRTC，而不是只用 WebSocket 传图片？

WebSocket 适合传控制消息和 JSON 结果，但持续传 JPEG/PNG 图像会有较大的编码和网络开销。WebRTC 更适合低延迟视频流，它可以用 H.264 编码走独立媒体通道，浏览器端原生支持播放。项目里 WebSocket 主要负责摄像头输入、信令和 JSON 数据，WebRTC 负责推理结果视频，这样职责更清晰。

Q36：TensorRT 在项目里解决什么问题？

TensorRT 主要解决 YOLO 推理延迟问题。YOLO 的 PyTorch 权重可以导出为 TensorRT engine，在固定 GPU 上运行时推理速度更快。README 里也给了导出命令和性能参考。对于实时会议系统，延迟和帧率很重要，所以 TensorRT 是从算法原型走向实时工程的一步。

Q37：recall boost 补漏检测是什么？

主检测模型是姿态模型时，背身、遮挡或者边缘区域的人可能关键点不稳定，导致 pose 检测漏掉。recall boost 会额外跑一个轻量纯检测模型，把这些没有关键点但有人形框的目标找回来，再和主检测结果做 IoU 去重融合。对于补漏出来的无关键点目标，系统用框顶附近合成一个头部参考点，继续做角度和扇区估计。

Q38：为什么要输出俯仰角和距离，不只输出水平角？

水平角能告诉音频阵列目标在哪个方向，但会议空间本质是 3D 的。俯仰角可以描述目标相对摄像头上下位置，距离可以大致判断人离设备多远。距离估计在项目里主要基于双眼关键点像素间距和标定关系，精度受姿态和关键点质量影响，但它能让系统输出从二维检测升级为更完整的空间感知结果。

Q39：MediaPipe 嘴部开合说话检测和音频 VAD 有什么区别？

MediaPipe 嘴部开合检测是视觉侧线索，通过嘴部关键点计算 MAR，判断嘴是否在张合。音频 VAD 是声音侧线索，判断当前音频里是否有人声。两者都不完美：张嘴不一定有声，有声也不一定看得见嘴；音频 VAD 也可能被噪声或其他人声音干扰。所以更合理的方式是把嘴部开合、视觉方位、音频 DOA 和 VAD 结合起来，提高 active speaker 判断的稳定性。

Q40：这个项目现在还有哪些明显不足？

第一，小人脸在低像素下 FaceID 仍然不稳定，只能通过质量过滤、TrackID 绑定和调试分析降低问题，不能完全消除。第二，视觉说话检测目前只是基于嘴部开合的辅助线索，还没有和真实麦克风阵列 DOA 做完整闭环。第三，距离估计依赖关键点质量和标定，在大角度侧脸或遮挡时会退化。第四，当前多模态融合更偏规则系统，后续可以在积累同步音视频数据后探索概率融合或一体化模型。

11. 最推荐的回答模板

如果面试官让你“介绍一下这个实习项目”，可以这样说：

我在思必驰做的是多模态会议软硬件预研，目标是验证会议终端中视觉和音频的协同感知。视觉侧我主要负责从鱼眼摄像头输入到实时 WebUI 输出的链路，包括 GPU 全景展开、YOLO 姿态/人脸检测、HybridSORT/BoT-SORT 跟踪、方位角和扇区估计，以及 AdaFace 动态 FaceID 建库。
项目里一个比较实际的问题是会议摄像头下人脸很小，embedding 不稳定，阈值低会把多人合并，阈值高又会把同一个人拆开。为了解决这个问题，我做了 bbox 尺寸过滤、TrackID-FaceID 深度绑定、同一 FaceID owner 约束，以及人脸 crop 和 embedding 的聚类可视化调试。
这个视觉链路后续是服务音频的：视觉侧输出每个人的方位角和 TrackID/FaceID，音频侧用麦克风阵列做 DOA 和波束形成，再经过语音增强和 FunASR 转写。两者融合后可以实现会议场景中的“谁在什么方向说话、说了什么”的端侧感知闭环。
